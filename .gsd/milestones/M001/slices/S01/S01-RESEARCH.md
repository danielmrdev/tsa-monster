# S01: Astro scaffold + i18n foundation — Research

**Date:** 2026-03-17
**Slice:** S01 — Astro scaffold + i18n foundation

## Summary

Fresh repo. This slice creates the entire Astro 6 project from scratch: scaffold, Tailwind v4 wiring, i18n config, content collections schema, all layouts and components, and the two page routes (`/` redirect + `/en/` shell). The key risk flagged in the roadmap — Tailwind v4 config API — is confirmed: it's a material departure from v3 and the approach is now verified. The content collections API also changed in Astro 6.

The two riskiest items are proven by documentation:
1. **Tailwind v4** uses `@tailwindcss/vite` as a Vite plugin (not `@astrojs/tailwind`), with `@import "tailwindcss"` in a CSS file. No `tailwind.config.js`, no `content` array. Custom theme tokens go in a `@theme {}` block inside the CSS file.
2. **Content collections** in Astro 6 use the new `loader: glob(...)` API (imported from `astro/loaders`). The old `type: 'content'` syntax is gone. `z` is imported from `astro/zod` (not `zod` directly).

Everything else is standard Astro: native i18n with `prefixDefaultLocale: true`, `getRelativeLocaleUrl()` from `astro:i18n`, `getCollection()` + `render()` from `astro:content` for dynamic routes, `@astrojs/sitemap` for sitemap generation.

## Recommendation

Use `pnpm create astro@latest` (minimal template, TypeScript strict) to scaffold the project, then wire in Tailwind v4 + MDX + sitemap manually. Build in strict sequence: scaffold → Tailwind verified rendering → i18n routes working → content collections schema → layouts/components → pages. Each step is a verifiable checkpoint before proceeding to the next.

Do not use `astro add tailwind` — it installs the legacy `@astrojs/tailwind` wrapper for Tailwind 3. Install `@tailwindcss/vite` directly and configure it in `vite.plugins`.

### astro.config.mjs shape

```js
import { defineConfig } from 'astro/config';
import tailwindcss from '@tailwindcss/vite';
import mdx from '@astrojs/mdx';
import sitemap from '@astrojs/sitemap';

export default defineConfig({
  output: 'static',
  site: 'https://tsa.monster',
  i18n: {
    defaultLocale: 'en',
    locales: ['en'],
    routing: {
      prefixDefaultLocale: true,
    },
  },
  integrations: [mdx(), sitemap()],
  vite: {
    plugins: [tailwindcss()],
  },
});
```

Note: `redirectToDefaultLocale` is **not** set here — the root redirect is handled by the client-side script in `src/pages/index.astro` (Decision D002). Astro's built-in `redirectToDefaultLocale` would conflict with `prefixDefaultLocale: true` behavior.

### Content collection schema (src/content/config.ts)

```ts
import { defineCollection } from 'astro:content';
import { glob } from 'astro/loaders';
import { z } from 'astro/zod';

const reviews = defineCollection({
  loader: glob({ pattern: '**/*.{md,mdx}', base: './src/content/reviews' }),
  schema: z.object({
    title: z.string(),
    description: z.string(),
    category: z.enum(['kitchen', 'outdoor', 'home', 'beauty']),
    date: z.coerce.date(),
    heroImage: z.string(),  // relative path under public/images/
    excerpt: z.string(),
    products: z.array(z.object({
      name: z.string(),
      rating: z.number().min(1).max(5),
      description: z.string(),
      affiliateUrl: z.string().optional(),
    })),
  }),
});

export const collections = { reviews };
```

Note: With the `glob` loader, `entry.id` is the file path relative to the base (e.g. `kitchen/best-air-fryers.mdx`). For routing, extract the slug with `entry.id.split('/').pop()?.replace(/\.mdx?$/, '')`. Category comes from `entry.data.category`.

### Dynamic route pattern (src/pages/en/[category]/[slug].astro)

```astro
---
import { getCollection, render } from 'astro:content';
import ArticleLayout from '../../../layouts/ArticleLayout.astro';

export async function getStaticPaths() {
  const entries = await getCollection('reviews');
  return entries.map(entry => ({
    params: {
      category: entry.data.category,
      slug: entry.id.split('/').pop()?.replace(/\.mdx?$/, '') ?? entry.id,
    },
    props: { entry },
  }));
}

const { entry } = Astro.props;
const { Content } = await render(entry);
---
<ArticleLayout title={entry.data.title} description={entry.data.description}>
  <Content />
</ArticleLayout>
```

### Build Order

1. **Scaffold + Tailwind rendering** — `pnpm create astro`, install deps, wire Vite plugin, create `global.css`, add one Tailwind class to `index.astro`, run `pnpm dev` to confirm styles render. This retires the Tailwind v4 risk.
2. **i18n config + page routes** — add i18n block to `astro.config.mjs`, create `src/i18n/config.ts`, create `src/pages/index.astro` (redirect script), create `src/pages/en/index.astro` (shell). Verify `/` redirects to `/en/` in browser, `/en/` returns 200.
3. **Content collections schema** — create `src/content/config.ts` and empty category dirs. No articles yet — just confirm `pnpm build` doesn't error on empty collection.
4. **Layouts + components** — BaseLayout, ArticleLayout, Header, Footer, AffiliateDisclosure, ArticleCard, ProductCard. These are templates S02 and S03 depend on; they must be buildable even with placeholder content.
5. **Dynamic route** — `src/pages/en/[category]/[slug].astro` with `getStaticPaths`. With empty collection this generates zero routes — that's fine for S01.
6. **Public files** — `robots.txt`, `_headers`. Simple, no ordering constraint.
7. **Final `pnpm build`** — confirm no errors, `dist/` contains `sitemap.xml`, `dist/en/index.html` exists.

### Verification Approach

After each step, one fast check:

- Step 1: `pnpm dev`, open `http://localhost:4321`, confirm styled content visible (Tailwind working)
- Step 2: `http://localhost:4321/` redirects to `/en/` in browser (check network tab), `/en/` returns shell HTML
- Step 3: `pnpm build` succeeds with empty collection (no type errors in config.ts)
- Step 4+5: `pnpm build` succeeds with all layout/component files present
- Final: `dist/sitemap-index.xml` or `dist/sitemap-0.xml` present; `dist/en/index.html` present

## Don't Hand-Roll

| Problem | Existing Solution | Why Use It |
|---------|------------------|------------|
| Sitemap generation | `@astrojs/sitemap` | Automatically discovers all static routes; outputs valid XML; supports i18n `alternateLinks`. |
| MDX rendering | `@astrojs/mdx` | Official Astro MDX integration; `render()` API returns `Content` component directly usable in templates. |
| Tailwind v4 + Vite | `@tailwindcss/vite` | Official Vite plugin; handles CSS source scanning automatically — no content glob config needed. |
| URL generation | `getRelativeLocaleUrl()` from `astro:i18n` | Handles locale prefix consistently; auto-updates when locales change. |

## Common Pitfalls

- **Using `astro add tailwind`** — installs the legacy `@astrojs/tailwind` wrapper which pulls Tailwind 3, not v4. Install `@tailwindcss/vite` and `tailwindcss` directly and add to `vite.plugins`.
- **Old content collections API** — `type: 'content'` / `type: 'data'` is Astro 4/5 syntax. Astro 6 requires `loader: glob(...)`. `z` must come from `astro/zod`, not `zod` directly.
- **`entry.slug` doesn't exist in Astro 6** — with glob loader, the slug is derived from `entry.id`. Parse it explicitly: `entry.id.split('/').pop()?.replace(/\.mdx?$/, '')`.
- **Missing `site` in astro.config** — `@astrojs/sitemap` throws if `site` is not set. Always set `site: 'https://tsa.monster'`.
- **global.css not imported** — Tailwind v4 requires `@import "tailwindcss"` in a CSS file imported by your layout (or page). Styles won't appear if only the plugin is registered.
- **`redirectToDefaultLocale` conflict** — don't set this in astro.config when using a custom client-side redirect script at `/`. Astro's built-in redirect at the root would interfere.

## Open Risks

- **`@astrojs/mdx` + Tailwind v4 typography** — MDX body content renders as plain HTML not covered by Tailwind's utility classes. Styled article bodies typically require `@tailwindcss/typography` plugin (`prose` classes). This is relevant for S03 (article content). Worth installing `@tailwindcss/typography` in S01 and adding `prose` to ArticleLayout so S03 doesn't hit a styling gap. Tailwind v4 plugin syntax: `@plugin "@tailwindcss/typography"` in the CSS file (not `plugins: [require(...)]` in config).
- **Empty collection `getStaticPaths`** — with zero articles in S01, the dynamic route generates zero paths. This is fine but verify `pnpm build` doesn't error on it.
